# E2E Web Search —— 端到端交付调研 Skill

## 本 skill 的核心决策：该不该搜？

在端到端交付场景下，搜索是**有成本**的：

- 消耗时间（用户在等）
- 结果可能被不相关信息淹没
- 中文互联网搜索质量参差不齐

所以本 skill 的第一条原则是：**能不搜就不搜**。

### 触发判断决策树

```
用户或 Agent 需要一个信息 → 判断：

1. 这个信息对当前交付决策关键吗？
   ├─ 否 → 不搜。继续当前对话。
   └─ 是 → 下一步

2. 这个信息是否是我已知的（训练数据里有，且稳定不变）？
   ├─ 是 → 不搜，直接说。
   └─ 否 → 下一步

3. 这个信息是否可能快速过时（产品动态、人事、价格、政策）？
   ├─ 是 → 必须搜。
   └─ 否 → 下一步

4. 这个信息是否需要高可信度来源（不能凭印象）？
   ├─ 是 → 必须搜。
   └─ 否 → 可以说"据我所知"后直接回答
```

## 何时必搜（强触发）

### 场景 1：用户问"XXX 是怎么做的"

用户在探索需求时经常会问"竞品怎么做的"、"行业标准是什么"。这类问题**必须搜**：

- "飞书的机器人话题 thread 机制是怎么实现的？"
- "Stripe 的订阅续费通知最佳实践是什么？"
- "Linear 的评论系统有什么特别设计？"

**强触发**，因为竞品动态快速变化，凭印象容易误导方案设计。

### 场景 2：技术选型调研

PRD 或方案设计阶段提到具体技术选型：

- "Redis Stream 和 Kafka 对于我们这种场景哪个合适？"
- "2026 年主流的前端框架性能对比"
- "gRPC 和 tRPC 的 trade-off"

**强触发**，技术生态变化快，训练数据里的"主流"可能已经过时。

### 场景 3：最新标准/规范/法规

- "GDPR 对用户行为数据追踪的最新要求"
- "OpenAPI 3.1 规范"
- "中国个人信息保护法对 App SDK 的要求"

**强触发**，法规和标准的**当前版本**最关键。

### 场景 4:人物或公司最新动态

- "Anthropic 最近发布了什么？"
- "XX 公司的 Y 产品还活着吗？"
- "Z 作者最近写了什么关于 agent 的文章"

**强触发**，时效性决定这类信息只能通过搜索获取。

## 何时不搜（弱触发或不触发）

### 场景 1：已有对话信息足够

如果用户已经在对话中提供了信息：

```
用户："我们的 QPS 大概是 1000，主要是读多写少的场景。"

Agent：[不需要搜"QPS 1000 的选型建议"]
      基于对话信息，直接推理即可。
```

### 场景 2：稳定的技术基础概念

- 什么是 TCP 三次握手
- 什么是 B+ 树索引
- 什么是 Kafka 消费者组

这类**稳定**的概念，不需要联网。

### 场景 3：业务判断题

- "我们应该先做 A 还是 B？"
- "这个需求值得做吗？"

这是**基于业务上下文的判断**，搜索没帮助。应该交给 `adversarial-qa` 和 `requirement-clarification`。

### 场景 4：内部系统问题

- "我们的 XX 服务是怎么设计的？"
- "公司的 CI/CD 流程怎么走？"

这应该搜**内部资源**（`bytedance-cloud-docs`、`feishu-cli-search`），而不是 Web。见"协同调用"章节。

## 搜索战术

### 战术 1：短查询优先

- ❌ "2026 年 Stripe 订阅续费通知的具体实现方式和最佳实践以及如何避免用户流失"
- ✅ "Stripe 订阅续费通知"

搜索引擎对长查询处理差。**2-4 个关键词**最优。

### 战术 2:一次一个主题

如果有多个主题要调研：

```
❌ 不好：一次搜 "Stripe Shopify Square 订阅续费通知对比"

✅ 好：
  搜索 1: "Stripe 订阅续费通知"
  搜索 2: "Shopify 订阅续费通知"
  搜索 3: "Square 订阅续费通知"
  然后整合
```

### 战术 3：时间敏感加时间词

查最新动态时加年份：

- "Anthropic Claude 2026 最新发布"
- "GPT 最新模型 2026"

**但注意**：搜到的结果要看**实际日期**，不是 URL 里的年份。很多网页会在 URL 加 "2026" 但内容是 2023 的。

### 战术 4:信源可信度分级

搜索结果里的信源，可信度从高到低：

1. **官方文档/博客**（anthropic.com、stripe.com、github.com/xxx）
2. **学术论文 / 标准文档**（arxiv、IETF RFC）
3. **权威技术媒体**（InfoQ、The Morning Paper）
4. **知名工程师个人博客**（有可查的社区身份）
5. **Medium / 知乎 / CSDN**（需要谨慎判断作者可信度）
6. **内容农场 / SEO 垃圾**（❌ 尽量不引用）

**引用时明确标注信源**，让用户知道信息来自哪里。

## 结果呈现规范

### 结构化输出

不要把搜索结果**原样贴**给用户。整合为**结构化摘要**：

```markdown
## 调研：[主题]

### 核心结论
[一段话直接回答问题]

### 关键细节
- [细节 1]（来源：[信源]）
- [细节 2]（来源：[信源]）

### 可信度评估
[这次调研的可信度如何，有什么局限]

### 参考链接
- [链接 1] - [简短描述]
- [链接 2] - [简短描述]
```

### 必须标注信源

**每个关键陈述都要标明信源**。用户需要能追溯。

### 矛盾信息的处理

如果多个信源说法矛盾：

```markdown
### 注意：信源矛盾

- [来源 A] 说：X
- [来源 B] 说：Y

可能原因：
- 时间差异（A 是 2024 年文章，B 是 2026 年）
- 场景不同（A 针对 ToB，B 针对 ToC）
- [其他]

建议：[给出你的判断和理由，或明确说"需要用户自己验证"]
```

**不要自己编造**来"调和"矛盾。

## 协同调用

### 何时该调本 skill，何时该调其他 skill

```
需要外部信息（行业/竞品/技术动态）
  → e2e-web-search（本 skill）

需要公司内部文档/方案
  → bytedance-cloud-docs

需要飞书历史讨论/文档
  → feishu-cli-search + feishu-cli-read

需要内部代码搜索
  → bytedance-codebase

需要服务接口定义
  → bytedance-bam
```

**混合场景示例**：

```
用户："我们想做个用户分层，行业和公司内部都查一下有没有类似方案。"

Agent 工作流：
  1. 调用 bytedance-cloud-docs 搜"用户分层"  # 先查内部
  2. 如果内部有成熟方案 → 直接用，不搜外部
  3. 如果内部没有或不够 → 调用 e2e-web-search 搜行业方案
  4. 整合呈现
```

**原则**：先内部（更相关、合规），再外部（补充视角）。

## 反 AI-slop 规范

### 禁止：搜索结果的 AI-slop 重述

很多 AI 会把搜索结果**改写**成更"流畅"的文字，但丢了关键细节。本 skill 要求**保留关键原文**：

```markdown
❌ AI-slop 式：
"根据调研，Stripe 的续费通知机制经过了精心设计，
通过多个触发点和用户友好的方式来提醒用户..."

✅ 好的整合：
根据 Stripe 官方文档（2026 updated），续费通知的触发点：
- 续费前 7 天：邮件 + Dashboard 提示
- 续费前 1 天：邮件（仅限年付订阅）
- 续费失败后 4 次 retry：每次间隔 24 小时

来源：https://stripe.com/docs/billing/subscriptions/xxx
```

### 禁止：基于搜索结果的过度推断

搜索到的信息是**事实**，你的**推断**要明确标记：

```markdown
❌ 不好：
"Stripe 的做法适合我们。"
（用户不知道这是你的推断还是 Stripe 说的）

✅ 好：
**事实**（来自 Stripe 文档）：Stripe 用 7 天 + 1 天两个通知点。
**我的推断**：这个节奏适合我们 ToB 场景，因为 [理由]。
```

## 失败处理

### 搜不到相关信息

明确告诉用户，**不要编造**：

```
我搜了 "[查询词]"，没有找到直接相关的信息。
可能原因：
- 问题太新/太细，网上还没讨论
- 查询词不准确

建议：
- 换个角度查：[建议查询词]
- 或者直接在 [社区 X / 论坛 Y] 问
- 或者我们先用已知信息决策，之后再补调研
```

### 信息过时

明确标注：

```
我搜到的最新信息是 [2024-XX-XX] 发布的。
现在是 2026 年，这个信息可能已经过时。

你可以：
- 先基于此信息做假设
- 到 [官方来源] 验证最新版本
```

### 信息敏感不宜搜

如果查询涉及敏感信息（用户隐私、竞品商业机密、非公开信息），明确拒绝：

```
这个查询涉及 [敏感点]，不建议从 Web 获取。
建议通过正式渠道（如直接联系对方、法务咨询）处理。
```

## 典型用法示例

### 示例 1：需求澄清中的调研

```
用户："我们想做个订阅续费通知，但不知道用户容忍度怎么样。"

Agent：
🔧 正在使用 `e2e-web-search` skill，调研行业做法。

[搜索: "SaaS 订阅续费通知频率 用户容忍度"]
[搜索: "Stripe Shopify 订阅续费通知最佳实践"]

## 调研：订阅续费通知频率

### 核心结论
行业主流做法是 **2-3 次通知**：
- T-7 天：第一次（友好提醒）
- T-1 天：第二次（紧急提醒）
- T+0 失败后：4 次重试 + 用户通知

### 依据
- Stripe 官方文档（2026）：2-3 次通知 + 重试
- Chargebee 最佳实践博客（2025）：超过 4 次通知显著增加投诉
- Baymard Institute 研究：7-day 提前通知能让 60%+ 用户主动续费

### 给我们的建议
基于以上数据和我们 ToB 的特性，建议采用 **T-14 + T-7 + T-1** 三次通知（ToB 客户决策链长）。

---
需要更深入调研哪个方向？
```

## 自检清单（每次搜索前）

- [ ] 这个搜索真的必要吗？（决策树走过）
- [ ] 查询词是否简短具体？
- [ ] 是否需要加时间词？
- [ ] 是否应该先查内部资源？
- [ ] 结果呈现是否结构化？
- [ ] 是否标注了信源？
- [ ] 是否区分了事实 vs 推断？
