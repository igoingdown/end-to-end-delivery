# PRD 模板库（按项目类型）

> 不同类型项目的 PRD 结构会略有侧重。本文件提供模板起点。
> **每个模板都要根据实际项目删改，不要全盘照抄**。

---

## 模板选择指南

```
项目类型判断流程：

是面向公司外部用户吗？
├─ 是 → ToB（企业客户）还是 ToC（终端用户）？
│     ├─ ToB → 用【ToB PRD 模板】
│     └─ ToC → 用【ToC PRD 模板】
└─ 否 → 内部项目
      ├─ 工具/效率类 → 用【内部工具 PRD 模板】
      ├─ 平台能力类 → 用【平台能力 PRD 模板】
      ├─ 数据/AI 类 → 用【数据 AI PRD 模板】
      └─ 业务流程变更 → 用【流程变更 PRD 模板】
```

---

## 【ToC PRD 模板】

**适用**：面向消费者的功能（App、H5、Web）。

**重点章节**：

1. **用户故事**
   - "作为 [用户画像]，我想要 [做什么]，以便 [达到什么目的]"
   - 多写 3-5 个典型故事，覆盖核心场景

2. **用户流程图**
   - 必须有：首次使用的流程 + 老用户的流程
   - 用 mermaid 画，不画文字描述

3. **数据指标**
   - 核心指标（北极星）：1 个
   - 支撑指标：3-5 个
   - 每个指标要有**基线 + 目标**

4. **A/B 测试计划**
   - 分流比例
   - 实验周期
   - 显著性判定
   - 退出条件（什么时候 ship / kill）

5. **用户研究**（可选）
   - 访谈样本量
   - 关键洞察

**砍掉**（ToC 不需要）：
- 详细的 API 设计（放到技术设计文档）
- 管理端需求（除非核心）

---

## 【ToB PRD 模板】

**适用**：面向企业客户的产品（SaaS、PaaS、开发者工具）。

**重点章节**：

1. **客户价值主张**
   - 一句话价值主张（能放到销售 slides 的那种）
   - 解决的客户问题（用客户原话）
   - 目标客户画像（ICP）

2. **竞品对比**
   - 我们做的 vs 竞品做的
   - 我们的差异化（3 条以内）
   - 客户为什么选我们

3. **商业模型**
   - 这个功能怎么影响付费？
   - 是基础版包含，还是高级版独占，还是 add-on？
   - 对 ARR / NRR 的影响预期

4. **销售可落地性**
   - 销售话术（一段 60 秒的 pitch）
   - POC 计划
   - 培训材料需求

5. **合规与安全**
   - 涉及数据分类（P1/P2/P3）
   - 区域合规（GDPR、CCPA、中国区等）
   - 客户审计要求（SOC2、ISO 27001）

6. **集成需求**
   - 客户侧 IT 对接要求
   - 支持的认证方式（SSO、API Key、OAuth）
   - 数据导入导出

**砍掉**（ToB 不常用）：
- 过度的 A/B 测试（客户数少）
- 过度的用户增长设计

## 【内部工具 PRD 模板】

**适用**：服务内部同事的工具（运营后台、数据看板、自动化脚本）。

**重点章节**：

1. **ROI 计算**
   - 替代的人工工作量：X 人日/月
   - 建设成本：Y 人月
   - 回本周期：Y × 20 / X 个月
   - **回本超过 18 个月 → 项目要打问号**

2. **目标用户清单**
   - 具体到部门/团队
   - 人数
   - 每人使用频次

3. **核心场景**
   - 日常场景（每天/每周都做）
   - 偶发场景（每月做几次）
   - 应急场景（一年一两次但很重要）

4. **权限矩阵**
   - 谁能看 / 谁能改 / 谁能删
   - 审批流

5. **替代方案分析**
   - 现有工具能不能满足？
   - 为什么要自建而不是买？

6. **长期维护计划**
   - 谁来维护
   - 下个季度/半年的迭代预期

**砍掉**（内部工具不需要）：
- 过度的 UI 设计（够用就行）
- 国际化（除非有海外同事用）
- 移动端（除非明确需求）

## 【平台能力 PRD 模板】

**适用**：给其他团队/业务方使用的平台能力（中台服务、SDK、共享组件）。

**重点章节**：

1. **接入方需求**
   - 明确的接入方清单（团队名 + 联系人）
   - 每个接入方的具体诉求（不是泛泛而谈）
   - "必须 v1 就支持的接入方" vs "v2+ 再支持的"

2. **抽象设计**
   - 哪些是**共性**抽象出来？
   - 哪些是**个性**留给接入方定制？
   - 抽象层次（别过度设计）

3. **接入体验**
   - 最少几步能接入？
   - SDK / API 文档需求
   - 是否需要自助接入流程？

4. **容量与 SLA**
   - 目标 QPS、RT、可用性
   - 容量规划（支持 N 个接入方）
   - 降级和熔断策略

5. **观测与运维**
   - 接入方能看到的 Dashboard
   - 平台自己的运维 Dashboard
   - 告警和值班

6. **版本管理**
   - 向后兼容策略
   - API 版本化
   - 老接入方迁移计划

**砍掉**（平台不需要）：
- 丰富的 UI（通常无 UI 或 UI 次要）
- 消费者视角的用户故事（你的用户是其他团队）

## 【数据 AI PRD 模板】

**适用**：AI 模型、数据能力、算法功能。

**重点章节**：

1. **问题定义**
   - 要解决什么具体问题？
   - 为什么 AI/数据能解决（不是规则能解决）？

2. **数据准备**
   - 训练数据来源
   - 数据量级
   - 标注方式和成本
   - 冷启动方案

3. **模型/算法选型**
   - 候选方案（至少 2 个）
   - 选型理由
   - 基线（baseline）是什么

4. **效果指标**
   - 离线指标（准确率、召回、AUC...）
   - 在线指标（CTR、CVR、留存...）
   - 效果验收标准（最低接受线）

5. **效果保障**
   - 人工审核节点
   - 兜底规则（AI 失效时的 fallback）
   - 失败 case 的丑陋程度评估

6. **迭代计划**
   - 模型如何迭代
   - 数据如何积累
   - 线上反馈如何回流

7. **伦理与风险**
   - 偏见检测
   - 幻觉控制
   - 内容合规

**砍掉**（数据 AI 不强调）：
- 复杂的用户 UI（通常是嵌入到现有产品）

## 【流程变更 PRD 模板】

**适用**：修改已有业务流程（而非新建功能）。

**重点章节**：

1. **当前流程**（必须画图）
   ```mermaid
   graph LR
     A[当前 step 1] --> B[当前 step 2]
     B --> C[当前 step 3]
   ```

2. **新流程**（必须画图）
   ```mermaid
   graph LR
     A[新 step 1] --> B[新 step 2]
     B --> C[新 step 3]
   ```

3. **Diff 清单**
   - 新增的：...
   - 删除的：...
   - 修改的：...
   - **不变的**：...（很重要，防止误改）

4. **兼容策略**
   - 老数据怎么办？
   - 老用户怎么办？
   - 老代码路径保留多久？

5. **迁移方案**
   - 数据迁移
   - 用户迁移
   - 系统切换（蓝绿 / 灰度）

6. **灰度和回滚**
   - 灰度节奏：1% → 10% → 50% → 100%
   - 每阶段的观测标准
   - 回滚条件和快速回滚方式

7. **沟通计划**
   - 谁需要被提前通知（客户、运营、客服）
   - 通知渠道和时间
   - FAQ 准备

**重点提示**：流程变更 PRD 比新建 PRD **更危险**，因为影响现有用户。

## 通用章节的好坏示例

### "一句话说明" 示例

❌ **坏**：
> "本 PRD 主要描述用户分层运营能力的建设，通过为不同用户提供差异化体验，助力业务增长。"

✅ **好**：
> "给运营同学一个自助配置用户分层规则的后台，替代现在手工分层的 Excel 流程。"

---

### "为什么做" 示例

❌ **坏**：
> "当前，用户分层是业务精细化运营的重要手段。通过对用户进行分层，可以更好地理解用户、服务用户，实现业务的可持续增长。"
>
> （注：全是空话，没有数据，没有具体痛点）

✅ **好**：
> **现状**：运营同学每周花 4 小时手工分层（导 Excel、跑公式、上传回系统）。一年 52 周 × 4 小时 = 208 人时浪费。
>
> **痛点**：
> - 慢：一次分层要半天，运营错过最佳时机
> - 错：上个月因为 Excel 公式错误，导致 3% 用户被错分，被客服投诉
> - 不可审计：没人知道某个用户"为什么"被分到这一层
>
> **不做的代价**：明年运营团队扩编 2 人，但主要时间还是耗在 Excel 上，新人感到价值低，招聘品牌受损。

---

### "验收标准" 示例

❌ **坏**：
> - 系统运行稳定
> - 性能良好
> - 用户可以顺畅使用

✅ **好**：
> **功能验收**：
> - [ ] 运营能在 UI 配置 RPN 分层规则（不需要写代码）
> - [ ] 配置保存后 30 秒内生效（全量用户重新分层）
> - [ ] 配置错误时，UI 明确提示错误行号和原因，不允许保存
>
> **性能验收**：
> - [ ] 分层计算：单次全量重分 100 万用户 ≤ 10 分钟
> - [ ] 配置保存：P99 ≤ 500ms
>
> **业务指标**：
> - [ ] 运营每次分层配置时长 从 4 小时 降到 ≤ 15 分钟
> - [ ] 统计窗口：上线后 4 周，样本 ≥ 10 次

## 模板使用提醒

1. **不要照抄章节**：每个模板都有"重点章节"，根据你项目实际，**删掉不适用的**比"填满"更重要。

2. **不要拼凑**：不要为了"看起来完整"而硬拼内容。宁可明确写"本项目不涉及"。

3. **字数不是指标**：好的 PRD 可能 800 字，烂的 PRD 可能 8000 字。关注**每字都在传递信息**。

4. **PRD 是活文档**：发布后仍要随决策变化更新。顶部的"最后更新日期"很重要。
