# E2E Progress Notify —— 进度通知

## 定位

端到端交付的**贯穿性 skill**：在主流程的每个关键节点主动发送飞书通知。

**不是独立阶段**，而是配合其他 skill 使用：
- PRD 定稿 → 通知 PM 和利益相关方
- BITS 任务创建 → 通知研发和 QA
- BOE 部署完成 → 通知 QA 去验证
- PPE 工单创建 → 通知审批人

---

## 通知策略

### 通知时机（何时触发）

主 Agent 应在以下节点**自动**调用本 skill：

| 阶段 | 通知时机 | 通知对象 | 消息类型 |
|---|---|---|---|
| PRD | 定稿时 | PM、Leader | 卡片 + PRD 链接 |
| 研发任务 | BITS 创建成功 | 研发、QA | 卡片 + 任务链接 |
| 代码改造 | 所有 MR 合入 | 研发 Leader | 简短文本 |
| 远端测试 | 测试失败 | 研发负责人 | 卡片 + 失败详情 |
| BOE 部署 | 部署完成 | QA | 卡片 + 验证链接 |
| PPE 工单 | 工单创建 | 审批人 | 卡片 + 工单链接 |
| 阶段阻塞 | 某 skill 返回 BLOCKED | 发起人 | 卡片 + 阻塞原因 |

### 通知粒度

**避免**：每个细节都通知（会被团队屏蔽）
**应该**：只在**真正需要对方做动作**时通知

---

## 消息格式

### 卡片优先（Interactive Card）

优先用飞书 Interactive 卡片，因为：
- 信息层次清晰（标题 / 正文 / 按钮）
- 支持@人
- 支持点击跳转
- 移动端显示友好

**卡片模板**（通过 `feishu-cli-msg` 发送）：

```json
{
  "type": "interactive",
  "header": {
    "title": "📋 用户分层需求 · PRD 定稿",
    "theme": "blue"
  },
  "body": [
    {
      "type": "text",
      "content": "项目「用户分层规则自助配置」的 PRD 已定稿。"
    },
    {
      "type": "fields",
      "fields": [
        { "label": "负责人", "value": "@zhangsan" },
        { "label": "计划交付", "value": "2026-05-15" },
        { "label": "涉及仓库", "value": "3 个" }
      ]
    }
  ],
  "actions": [
    { "text": "查看 PRD", "url": "https://..." },
    { "text": "查看任务", "url": "https://..." }
  ]
}
```

### 文本兜底

卡片不适用的场景（简短进度更新）用文本：

```
@zhangsan 用户分层项目：所有 3 个 MR 已合入，即将进入 BOE 部署。
```

---

## 调用方式

**调用 `feishu-cli-msg`**：

```
让 feishu-cli-msg skill 发送卡片消息到：
- 目标：话题 ID / 群 ID / 个人
- 内容：卡片 JSON（按上面模板）
```

不要自己拼接 feishu OpenAPI 请求，让 `feishu-cli-msg` 处理认证、格式、重试。

---

## 上下文感知

本 skill 的价值在于**减少人工告知的工作量**，所以要**自动提取**当前上下文：

从主 Agent 的对话中提取：
- 项目名称（从 PRD 一句话）
- 负责人（从 `e2e-dev-task-setup`）
- 当前阶段
- 关键链接（PRD、BITS、MR 等）

**不要问用户**"通知谁？说什么？"——这失去了本 skill 的意义。

**少数情况**才问用户：
- @人不明确（PRD 没写）
- 通知到群还是私信不清楚

## 通知对象的确定

### 默认规则

- **PRD 定稿** → @PRD 里标的"审阅人"
- **研发任务创建** → @BITS 任务的 RD + QA
- **MR 合入** → @PRD 里的 RD Leader
- **BOE 部署完成** → @QA
- **PPE 工单创建** → @审批人
- **项目阻塞** → @项目发起人

### 话题 vs 群 vs 私信

- **阶段性大节点**（PRD、部署）→ 发到**话题**（公开）
- **需要立即响应**（阻塞、审批）→ 发到**私信**（私密）
- **周期性状态**（每日进度）→ 发到**群**（团队）

### 避免打扰

- 不在**下班时间**发非紧急通知（默认 20:00-9:00 静默）
- 不在**周末**发非紧急通知
- 紧急通知（阻塞、失败）不受时间限制，但附加歉意语

实现方式：通过 `feishu-cli-msg` 的 `scheduled_time` 延迟发送。

---

## 卡片主题色

按紧急程度选 theme：

| Theme | 适用场景 |
|---|---|
| `blue` | 一般进度更新（PRD 定稿、任务创建） |
| `green` | 成功类（部署完成、测试通过） |
| `orange` | 警告类（有 concerns、需要验证） |
| `red` | 阻塞/失败类（部署失败、工单被拒） |
| `grey` | 归档/总结类 |

## 和其他 skill 的协同

### 被谁调用

**几乎所有其他 skill**都可能在完成后调用本 skill：

```
e2e-dev-task-setup 成功 → 调 e2e-progress-notify("研发任务已创建")
e2e-deploy-pipeline 完成 BOE → 调 e2e-progress-notify("BOE 部署完成，请 QA 验证")
任何 skill 返回 BLOCKED → 调 e2e-progress-notify("项目被阻塞：XXX")
```

### 调用 feishu-cli-msg

单一依赖：`feishu-cli-msg`。

**不要**自己调飞书 API。

---

## 失败处理

### 失败 A：feishu 登录过期

- 调 `feishu-cli-auth` 重新登录
- 重试

### 失败 B：目标群/话题不存在

- 告知主 Agent：通知发送失败
- 主 Agent 决定：继续流程 + 口头提醒用户手动通知

### 失败 C：卡片格式错误

- `feishu-cli-msg` 返回格式校验失败
- Fallback 到文本消息

### 核心原则

通知失败**不阻塞**主流程。流程已经走完，通知发不出去不影响实际结果。

## 反 AI-slop 规范

### 禁用模式

- ❌ 每个细节都通知（变成噪音）
- ❌ 卡片内容空泛（"项目有新进展"）
- ❌ @全部人（骚扰）
- ❌ 半夜发非紧急通知
- ❌ 通知里加"请及时响应"这种废话

### 正确模式

- ✅ 只在**对方需要做事**时通知
- ✅ 卡片直接说明白：做了什么、接下来要做什么、对方要做什么
- ✅ 精准@相关人
- ✅ 非紧急时尊重对方作息
