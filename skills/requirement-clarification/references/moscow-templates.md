# MoSCoW 模板库（按项目类型）

> 不同类型项目的 Must/Should/Could/Won't 常见分布。用作起始模板，**不要照抄**——每个项目都有特殊性。

---

## ToC 功能模板

### Must have（典型）

- 核心功能闭环（用户能完成主要任务）
- 登录/认证链路
- 基础的异常处理（网络错误、空状态）
- 核心路径的性能保证（响应时间 < 3s）
- 基础数据埋点（能知道有人在用）

### Should have（典型）

- 核心路径的 A/B 能力
- 推送/通知联动
- 分享能力
- 历史记录/收藏
- 简单的个性化（基于用户等级、偏好）

### Could have（典型）

- 动画、过渡效果的精细化
- 深度个性化（推荐算法）
- 国际化（多语言、多 Region）
- 无障碍适配
- 高级设置项

### Won't have（典型）

- 第三方账号登录（如果不是核心场景）
- 离线能力（除非明确需要）
- 多端实时同步（如果单端能跑）
- AI 生成内容（除非核心卖点）

## 内部工具 / 效率工具模板

### Must have

- 核心业务流程可跑通（替代原来的人肉操作）
- 基础权限（谁能看谁能改）
- 操作日志（出了问题能查）
- 数据导出（Excel/CSV）

### Should have

- 批量操作（选多个对象一起处理）
- 历史版本和回滚
- 更细粒度的权限（部门、角色）
- 告警通知（关键操作通知相关人）
- 搜索和过滤

### Could have

- 数据可视化（图表、Dashboard）
- 自定义视图
- 工作流审批
- 开放 API 给其他系统

### Won't have

- 移动端（除非 70% 用户在手机上用）
- 外部用户支持（只服务内部）
- 极致性能优化（内部用户容忍度高）

---

## 平台能力 / 中台模板

### Must have

- 核心能力（你这个平台存在的理由）
- SDK / API 接入能力
- 基础的接入方认证和鉴权
- 最基础的观测（接入方量、调用量）
- 接入文档

### Should have

- 自助接入流程（不需要找平台同学）
- 容量管理和限流
- 多租户隔离
- 故障自动降级
- 接入方数据 Dashboard

### Could have

- 高级定制能力
- Plugin 扩展机制
- 多 Region 部署
- 蓝绿部署 / 金丝雀
- 复杂的 SLO 保障

### Won't have

- 全自动化运维（v1 允许人工介入）
- 跨云/跨公司部署
- 极致性能（先对齐业务 SLA 即可）

---

## 数据 / AI 能力模板

### Must have

- 核心模型或规则能跑
- 离线数据准备（训练数据或规则数据）
- 基础效果评估（准确率、召回率、覆盖率）
- 兜底逻辑（模型失效时的降级）

### Should have

- 在线 A/B 测试能力
- 模型版本管理和回滚
- 数据漂移检测
- Shadow 模式（不生效但跑预测）
- 结果解释性（为什么是这个结果）

### Could have

- 多模型 Ensemble
- 自动重训练
- 主动学习（用户反馈循环）
- 跨场景迁移

### Won't have

- 极致模型效果（v1 先跑通）
- 实时训练（延迟 > 几小时是 OK 的）
- 多模态（除非核心诉求）

---

## 业务流程变更模板（非新建）

### Must have

- 新流程跑通
- 老流程兼容（老用户不炸）
- 数据迁移方案
- 灰度能力

### Should have

- 监控告警覆盖所有变更点
- 快速回滚能力
- 变更前后对比报表
- 客服/运营侧的培训和文档

### Could have

- 灰度更细粒度（按用户画像）
- A/B 对比新旧流程效果
- 双写兼容期延长

### Won't have

- 一次切 100%（务必灰度）
- 删除老流程代码（留 3 个月再删）

## 模板使用说明

### 第一步：选模板

根据你的项目类型从上面选一个**最接近**的模板。

### 第二步：勾选适用项

模板里的每一条，用户要回答三个选项之一：
- ✅ 这项我们要做
- ❌ 这项不适用我们
- ❓ 不确定，需要讨论

### 第三步：加自己的

模板覆盖通用项。**你的项目特殊需求**要额外补充。

### 第四步：重新分级

勾选 + 补充完成后，**重新看一遍 Must**：
- Must 超过 60% → 降级部分到 Should
- Must 数量 > 10 条 → 砍到 10 以内

### 第五步：验证分级合理性

逐条问："如果 v1 不做，能上线吗？"——用这个问题卡住每个 Must。

## 混合项目类型

很多项目是**混合型**（比如"面向内部运营的、带 AI 能力的平台能力"）。这种情况下：

1. 识别主类型（占比最大的那类）
2. 选对应模板作为基础
3. 从其他相关模板挑额外条目补充

例子："面向运营的 AI 推荐能力"
- 主类型：内部工具（运营用）
- 辅助类型：AI 能力 + 平台能力
- 拼装：以"内部工具"为骨架，加"AI 能力"的模型管理项 + "平台能力"的多租户隔离

## 反 AI-slop：MoSCoW 常见错误

### 错误 1：Must 无限多

```
❌ 错误：
M1: 功能可用
M2: 用户体验好
M3: 性能稳定
M4: 安全可靠
M5: 界面美观
...（还有 20 条）

(几乎没有 Should，Could，Won't)
```

**问题**：这不是分级，是口号。

**修复**：砍到 5 条以内的真 Must，其他降级或删除。

### 错误 2：Should 和 Could 不分

Should 和 Could 的界限经常模糊。用**承诺度**区分：

- Should：**我承诺 v1.1 会做**
- Could：**做不做两说**

如果你说不清具体哪个版本做，就是 Could。

### 错误 3：Won't 只写一条"未来再说"

```
❌ 错误：
W1: 未来再考虑的功能

✅ 正确：
W1: 国际化（理由：翻译成本高，先验证核心价值）
W2: 自动化分层（理由：底层画像数据还没对齐）
W3: 超过 10 层的分层（理由：场景频率极低）
```

**原则**：Won't 要有**具体**的"不做什么 + 为什么"，而不是兜底说"未来再看"。
