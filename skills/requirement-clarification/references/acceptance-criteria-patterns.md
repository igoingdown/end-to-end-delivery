# 验收标准样例库

> 好的验收标准 = **可测、可量化、无歧义**。
> 本文档给出不同场景的验收标准模板，作为撰写参考。

---

## 基本格式

验收标准用 **Given-When-Then**（给定-当-则）三段式。不需要机械照抄，但要保持**可验证性**。

```
给定 [前置条件和系统状态]
当 [用户或系统的某个动作]
则 [明确的预期结果]
```

---

## 模板一：用户功能验收

### 好的例子

```markdown
### 功能：用户分层查看

给定 用户已登录且已完成手机验证
当 用户点击"我的分层"入口
则 预期：
- 3 秒内看到本人所属的分层卡片
- 卡片上显示：分层名称、升级所需条件、升级进度条
- 如果用户未达到任何分层，显示"暂未分层"引导卡片
```

### 坏的例子（为什么不行）

```markdown
❌ 用户可以查看自己的分层信息，界面简洁美观
```

**问题**：
- "简洁美观" 不可测
- 没说前置条件
- 没说展示什么具体内容
- 没说时间约束

## 模板二：异常场景验收

**每个主流程都必须有至少 3 个异常场景的验收**。

```markdown
### 异常：网络失败

给定 用户点击"我的分层"时，网络请求失败（超时或 5xx）
当 重试 2 次仍失败
则 预期：
- 显示"无法加载，请稍后再试"的友好提示
- 提供"手动重试"按钮
- 不崩溃，不留白屏
- 错误上报到 Sentry / APM，能被运维同学发现

---

### 异常：空状态

给定 用户是新注册用户（0 行为数据）
当 用户打开"我的分层"
则 预期：
- 显示"暂无分层数据"空状态页
- 说明为什么（"需要使用 3 次后才会分层"）
- 引导下一步行动（"去首页逛逛"）
- **不显示**数据 loading 菊花，因为没有数据可加载

---

### 异常：权限不足

给定 用户 token 过期
当 请求分层接口返回 401
则 预期：
- 跳转到登录页
- 登录成功后回到原页面
- **不展示** token 过期的具体错误信息给用户（安全考虑）
```

## 模板三：性能验收

```markdown
### 性能：接口响应时间

接口：GET /api/user/level

- P50 ≤ 100ms
- P99 ≤ 300ms
- 错误率 < 0.5%

统计窗口：上线后连续 7 天
采样：每分钟至少 10 次调用

压测标准：1000 QPS 持续 10 分钟，P99 ≤ 500ms 不崩溃

---

### 性能：页面加载

首屏渲染（FCP）：
- 4G 网络 ≤ 1.5s
- Wi-Fi 网络 ≤ 0.8s

可交互时间（TTI）：
- 4G 网络 ≤ 3s
- Wi-Fi 网络 ≤ 1.5s
```

## 模板四：安全验收

```markdown
### 安全：数据访问控制

- 用户只能查看**自己的**分层数据
- 尝试通过 URL 参数查询他人数据时：
  - 接口返回 403
  - 前端显示"无权访问"
  - 服务端记录安全日志
- 批量数据导出接口需要额外的 admin 权限校验

---

### 安全：敏感信息脱敏

- 分层规则中涉及用户手机号、身份证等字段时：
  - 前端展示：手机号 138****1234
  - 日志输出：完全脱敏（不留明文）
  - 导出文件：根据导出人权限决定是否脱敏

---

### 安全：SQL 注入防护

分层规则的 DSL 输入经过严格的解析器处理，不拼接原始 SQL。
  测试用例：
  - 输入 `'; DROP TABLE users; --`
  - 预期：被解析器拒绝，返回"规则语法错误"
```

## 模板五：兼容性验收

```markdown
### 客户端兼容

- iOS：支持 iOS 14+（覆盖 > 98% 用户）
- Android：支持 Android 8+（覆盖 > 95% 用户）
- Web：Chrome 90+, Safari 14+, Edge 90+，**不支持 IE**

### 向后兼容

- 老版本客户端（v2.x）请求新接口时：
  - 新字段缺失时返回默认值（不报错）
  - 字段类型兼容（如 int → long 是兼容的）
- 接口版本策略：v1 保留至少 6 个月

### 数据兼容

- 老数据（schema v1）能被新代码读取
- 新数据（schema v2）要在 dry-run 阶段验证老客户端能读
```

## 模板六：业务指标验收

最难但最重要的一类。必须**可量化 + 有时间窗 + 有样本要求**。

```markdown
### 关键业务指标：运营效率

指标：分层配置时长
- 基线：运营同学手工分层平均 4 小时 / 次
- 目标：上线后平均 ≤ 15 分钟 / 次

统计方式：
- 时间窗：上线后第 2 周 - 第 5 周
- 样本量：≥ 10 次分层配置操作
- 统计方法：中位数（避免极值影响）

达成判定：
- ✅ 达成：中位数 ≤ 15 分钟
- ⚠️ 部分达成：中位数 16-30 分钟，需要 root cause 分析
- ❌ 未达成：中位数 > 30 分钟，需要迭代优化

---

### 关键业务指标：运营准确率

指标：分层错误率（错分 / 总分层操作）
- 基线：人工分层错误率约 5%
- 目标：系统分层错误率 ≤ 1%

统计方式：
- 时间窗：上线后连续 30 天
- 采样：人工抽检 10% 的分层结果
- 反馈渠道：运营同学的 bug 反馈
```

## 模板七：数据埋点和可观测验收

```markdown
### 埋点清单

功能必须埋点：
- 页面曝光（PV）
- 核心按钮点击（分层查看、分层升级）
- 异常展示（空状态、网络错误）
- 退出节点（用户从哪一步放弃）

每个埋点必须包含：
- event_name：事件名
- user_id：用户 ID（可脱敏）
- timestamp：时间戳
- context：场景上下文（分层 ID、来源页）

---

### 监控告警

必须配置的告警：
- [ ] 接口 5xx 率 > 1%，持续 3 分钟 → P1 告警
- [ ] P99 响应时间 > 500ms，持续 5 分钟 → P2 告警
- [ ] 用户分层失败率 > 2% → P2 告警
- [ ] 数据源异常（上游分层计算服务挂了）→ P0 告警

告警接收方：
- P0/P1：值班同学 + 负责人
- P2：值班同学

---

### Dashboard

上线前必须准备：
- [ ] 功能健康度 Dashboard（QPS、RT、错误率）
- [ ] 业务指标 Dashboard（分层分布、升级率）
- [ ] 用户漏斗 Dashboard（从入口到分层查看的转化）
```

## 模板八：灰度和发布验收

```markdown
### 灰度计划

第一阶段：内部灰度
- 范围：字节员工
- 时长：3 天
- 验收：无 P0 bug，核心指标符合预期

第二阶段：1% 灰度
- 范围：随机 1% 用户
- 时长：3 天
- 验收：错误率 < 基线的 1.2 倍

第三阶段：10% 灰度
- 范围：随机 10% 用户
- 时长：7 天
- 验收：核心业务指标达成预期的 80%

第四阶段：50% 灰度
- 时长：7 天

第五阶段：100% 全量
- 触发条件：前阶段无异常

---

### 回滚标准

触发自动回滚：
- 5xx 错误率 > 5%，持续 5 分钟
- 关键接口 P99 > 1s，持续 10 分钟

触发手动回滚评估：
- 用户投诉 > N 条
- 业务指标明显下降
- 数据异常（上下游不一致）

回滚时间目标：
- 配置变更类：5 分钟内
- 代码发布类：30 分钟内
```

## 反 AI-slop：验收标准常见错误

### 错误 1：不可测

```
❌ "用户体验流畅自然"
❌ "系统稳定可靠"
❌ "界面美观大方"

✅ "首屏加载 ≤ 1.5s（4G 网络）"
✅ "连续运行 30 天无 P0 故障"
✅ "点击热区 ≥ 44pt（iOS HIG 标准）"
```

### 错误 2：没有数字

```
❌ "响应时间要快"
❌ "支持大量用户"

✅ "P99 ≤ 200ms"
✅ "支持 10K QPS（峰值）"
```

### 错误 3：没有时间窗

```
❌ "分层准确率 ≥ 95%"

✅ "分层准确率 ≥ 95%（统计窗口：上线后 30 天，人工抽检 10%）"
```

### 错误 4：一条验收包含太多断言

```
❌ 一条验收：
"用户可以查看、编辑、删除分层规则，支持批量操作，操作后实时生效，性能好，有审计日志"

✅ 拆成多条：
- 查看功能验收
- 编辑功能验收
- 删除功能验收
- 批量操作验收
- 实时生效验收（性能类）
- 审计日志验收（安全类）
```

## 快速清单：验收标准最少覆盖

每个需求的验收标准必须至少覆盖以下 5 类（即使某类只有 1 条）：

- [ ] **功能验收**：主流程 Given-When-Then
- [ ] **异常验收**：至少 3 个异常场景
- [ ] **性能验收**：至少 1 个数字化目标
- [ ] **安全验收**：至少 1 条（即便是"仅限本人访问"这种基础项）
- [ ] **数据验收**：至少 1 个业务指标或埋点要求

缺失任何一类 → 需求定义不完整。

*本样例库基于字节内部真实验收场景整理。如发现新的验收模式 → 补充进来。*
