# Requirement Clarification —— 结构化需求澄清

## 本 skill 和 adversarial-qa 的分工

| 维度 | adversarial-qa | requirement-clarification |
|---|---|---|
| 目的 | 砍掉不合理需求 | 组织合理需求为结构化清单 |
| 语气 | 挑战、找茬 | 协助、整理 |
| 输入 | 模糊想法 | 核心方向已稳定 |
| 输出 | 稳定的核心需求 | **MoSCoW 清单 + 边界 + 验收标准** |

**切换时机**：`adversarial-qa` 触发 HARD-GATE "核心需求已稳定"后，本 skill 接手。

---

## 核心产出物

本 skill 完成后必须产出**三份结构化文档**（作为 `prd-generation` 的输入）：

### 产出物 1：MoSCoW 需求清单

```markdown
# 需求清单（MoSCoW）

## M - Must have（必须做，不做则项目失败）
- [M1] ...
- [M2] ...

## S - Should have（应该做，做了显著提升价值，不做也能 launch）
- [S1] ...
- [S2] ...

## C - Could have（可以做，锦上添花）
- [C1] ...

## W - Won't have（明确不做）
- [W1] ...
- [W2] ...
```

### 产出物 2：边界说明

```markdown
# 边界定义

## 覆盖场景
- 场景 1：[描述]
- 场景 2：[描述]

## 明确不覆盖
- ❌ [场景]：[为什么不做]
- ❌ [场景]：[为什么不做]

## 与现有功能的关系
- 和 [功能 A] 的边界：[差异和衔接]
- 和 [功能 B] 的关系：[...]
```

### 产出物 3：验收标准

```markdown
# 验收标准

## 功能验收
- [ ] [功能点 1]：给定 [输入]，操作 [动作]，预期 [结果]
- [ ] ...

## 非功能验收
- [ ] 性能：[指标]
- [ ] 安全：[要求]
- [ ] 兼容性：[范围]

## 数据验收
- [ ] 关键指标：[指标] 提升 [数值] 在 [时间窗]
```

## 工作流程

### 阶段 A：拆解功能点（一次一个维度）

从 `adversarial-qa` 交接的"核心需求"通常是**一句话**。你的第一步是**展开**为功能点清单。

采用**维度拆解法**：

```
询问用户（一次一个维度）：

维度 1 - 用户流程：
"从用户视角，走完这个功能他要做哪几步？"

维度 2 - 数据流：
"数据从哪来？到哪去？中间要做哪些加工？"

维度 3 - 异常场景：
"失败 / 为空 / 超时 / 权限不足 时分别怎么办？"

维度 4 - 管理端：
"运营 / 客服 / 数据团队 要不要看、改、查这个功能的数据？"

维度 5 - 边界交互：
"和现有的 X 功能有没有冲突或联动？"
```

每问一个维度，把用户的回答拆成具体的功能点。

### 阶段 B：MoSCoW 分类

拿到功能点清单后，逐条分类。

**分类的标准提问**：

```
对每一条功能点，问：
"这个功能点如果 v1 不做，能上线吗？"
  ├─► 不能 → Must
  └─► 能，但用户体验差 → Should
       │
       └─► "如果 v1 不做，v2 再做，会造成什么问题？"
              ├─► 有明显问题 → Should
              ├─► 没明显问题但有用户会抱怨 → Could
              └─► 没影响 → Won't
```

**关键陷阱警告**：

- ⚠️ **用户倾向把所有东西标 Must**。你的工作是**逼他承认某些是 Should/Could**。
- ⚠️ **Must 类项超过 60% → 重新分类**。正常分布应该是 Must ≤ 50%。

**战术**："如果只能做 3 个功能点，你选哪 3 个？"——选出的就是 Must，剩下的重新排。

### 阶段 C：边界定义

用**正反两面**定义边界：

**正面**：我做什么？
- 列出覆盖的场景
- 列出支持的用户群
- 列出支持的数据类型

**反面**：我明确不做什么？（⭐ 这条更重要）
- 列出**明显应该做但现在不做**的场景（并说明原因）
- 列出**用户会误以为支持但实际不支持**的场景（防止预期错位）

**边界示例**：

```
❌ 不做：国际化（v1 只支持简体中文环境）
  原因：翻译成本高，先验证核心价值

❌ 不做：自动分层（只做规则配置 + 手动触发）
  原因：自动化需要用户画像数据还没对齐

❌ 不做：支持超过 10 层的分层
  原因：超过 10 层的运营场景频率极低，后续按需支持
```

### 阶段 D：撰写验收标准

**三类验收必须有**：

1. **功能验收**：给定-操作-预期的三段式
   ```
   给定：登录用户且已经完成手机验证
   操作：点击"分层查看"
   预期：看到本人所属的分层卡片
   ```

2. **非功能验收**：性能 / 安全 / 兼容
   ```
   性能：P99 响应时间 ≤ 200ms
   安全：只能看自己的分层信息，不能查他人
   兼容：支持 iOS 14+ 和 Android 8+
   ```

3. **数据验收**：可量化的业务指标
   ```
   关键指标：运营分层配置时长 从 平均 4 小时/次 降低到 15 分钟/次
   统计窗口：上线后 4 周内
   样本要求：至少 10 次分层配置操作
   ```

## 提问战术

### 战术 1：框架化 vs 开放式

**禁忌**：一次抛空白问题让用户自由发挥。用户会懵，或者给出无信息量的答案。

**正确**：给出**框架**让用户在框架里回答。

```
❌ 不好："这个功能还有什么细节要考虑？"

✅ 好：
"我们还差以下细节：
  1. 用户第一次用的引导流程
  2. 网络失败的降级方案
  3. 数据异常时的展示

你想先聊哪一个？"
```

### 战术 2：用模板降低用户认知成本

给用户一个可以直接"填空"的模板：

```
场景验收模板：
  给定 _______________
  操作 _______________
  预期 _______________

你试试填一个关键场景？
```

### 战术 3：主动补充用户没想到的

很多用户不会主动想到：

- 异常流程（网络断、数据空、权限不足）
- 管理端需求（运营、客服、数据）
- 监控和告警（上线后怎么知道健康）
- 灰度策略（怎么分步上线）
- 迁移方案（老数据怎么办）

当用户说"我觉得差不多了"时，**主动提醒**这些维度，问"你考虑过吗"。

当**所有三份产出物都具备**时，触发 HARD-GATE：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
<HARD-GATE>

结构化需求澄清完成。产出：

【MoSCoW 清单】
M: X 条 | S: Y 条 | C: Z 条 | W: W 条

【边界定义】
覆盖 X 个场景 | 明确不做 Y 个场景

【验收标准】
功能验收 X 条 | 非功能 Y 条 | 数据 Z 条

以上是否准确反映了你的需求？
- ✅ 确认 → 进入 PRD 生成阶段

</HARD-GATE>
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 反 AI-slop 规范

### 禁用模式

- ❌ 把 MoSCoW 填得无信息量：`[M1] 用户可以使用本功能` —— 废话
- ❌ 边界定义写成"本功能覆盖用户分层管理的核心场景"—— 抽象
- ❌ 验收标准写成"性能良好，用户体验好"—— 不可测

### 正确模式

- ✅ `[M1] 运营同学能通过 UI 配置分层规则（RPN 表达式），不需要找 RD 改代码`
- ✅ `❌ 不做：跨 App 分层（只支持本 App 内用户），原因：数据跨 App 合规待评审`
- ✅ `性能：P99 ≤ 200ms，统计窗口 5 分钟，采样 100 次以上`

## 子场景：不同项目类型的差异

不同类型项目的 MoSCoW 侧重不同：

### ToC 功能
- **Must** 偏向"核心体验闭环"（能用起来）
- **Should** 偏向"留存要素"（复购、分享、通知）
- **Could** 偏向"锦上添花"（个性化、动画）

### 内部工具
- **Must** 偏向"业务流程能跑通"
- **Should** 偏向"批量操作、权限管理"
- **Could** 偏向"数据分析、导出"

### 平台能力
- **Must** 偏向"接入方真实诉求"
- **Should** 偏向"可观测、可运维"
- **Could** 偏向"高级定制能力"

见 `references/moscow-templates.md` 按项目类型查模板。
