# 对抗式问答 · 问题库

> 按业务类型分类的问题库。遇到具体场景时参考对应章节，不要把全部问题一次抛出。

---

## 通用基础问题（任何需求都要问）

### 必要性类

- 不做这个会怎样？
- 现在用户/业务方是**怎么**没有这个功能也活下来的？
- 过去 N 年为什么没人做？为什么现在要做？
- 这是"真需求"还是"想做"？需求来源是谁？
- 如果只剩 3 个需求能做，这个会进前 3 吗？

### 优先级类

- 这个功能**不做的话**业务 KPI 会减少多少？
- 如果做了，预期收益是多少？数据假设是什么？
- 用户会愿意为这个功能付出额外成本吗（操作、学习、付费）？

### 风险类

- 这个功能最坏的情况是什么？
- 做错了怎么回滚？
- 下游依赖如果挂了怎么办？

---

## ToB（企业产品）场景

**特点**：决策路径长、客户画像清晰、价值可量化。

### 客户价值

- 哪些客户会付钱用这个功能？ARR 能提升多少？
- 客户流失风险中，有多少是因为缺这个功能？有数据吗？
- 有没有 Top 10 客户明确说过"没有这个不用"？
- 竞品有吗？有的话为什么客户还没走？没的话为什么我们要做？

### 销售可落地性

- 销售团队怎么卖这个功能？一句话卖点是什么？
- POC 怎么做？多久能让客户感知到价值？
- 需要培训多久客户才能用起来？

### 合规风险

- 涉及什么数据？客户会不会不让我们碰？
- 国际化场景下有哪些法规限制（GDPR / CCPA）？
- SOC2、ISO 27001 影响？

---

## ToC（消费者产品）场景

**特点**：用户基数大、指标导向、心智易变。

### 用户心智

- 用户**现在**是怎么解决这个问题的？（不是"如果有了会怎么用"）
- 这个功能解决的是"必须解决"、"解决了更好"、还是"用户根本没觉得是问题"？
- 用户调研了吗？调研样本多少？有没有明显偏差（只问了忠诚用户）？

### 数据假设

- DAU / MAU 影响预期是多少？基于什么假设？
- 留存曲线会怎么变？7日、30日留存？
- 转化漏斗在哪个环节？提升多少？
- 有没有 A/B 测试计划？分流比例？显著性怎么算？

### 竞品验证

- 微信、抖音、小红书有类似功能吗？他们怎么做的？
- 他们**为什么**不做？/ 他们**为什么**做了又下线了？
- 我们的差异化在哪？凭什么用户会选我们的版本？

---

## 内部工具 / 效率类需求

**特点**：用户是内部同事、ROI 容易算、需要警惕"过度工程"。

### 真实 ROI

- 目标用户群（内部）有多少人？
- 每人每次用这个工具节省多少时间？每月用几次？
- 总节省：人数 × 次数 × 时间 = X 人日/月
- 建设成本：Y 人月 → 回本要多久？
- **如果回本超过 18 个月，这个项目可能不值得做**

### 替代方案

- 现有工具（飞书文档、在线表格、内部已有平台）能满足 80% 吗？
- 写个 Excel 宏 / 脚本能不能临时顶一下？
- 是不是在重复造轮子？**全公司搜过了吗**？

### 长期维护

- 上线后谁来维护？下个季度还有资源吗？
- 如果负责人离职了，这个工具会不会就死了？
- 有没有人会主动迭代？

---

## 平台能力类需求（给其他团队用的能力）

**特点**：接入方众多、抽象层次高、容易过度设计。

### 接入方真实诉求

- 有几个**具体**的接入方说"我们需要这个"？不是"可能用得上"。
- 这 N 个接入方的需求有多少是共性的？多少是个性化的？
- 平台能力是"最大公约数"还是"最小公倍数"？

### 抽象层次

- 这个抽象会不会太早？先让 2-3 个业务方各自实现，再抽象会不会更好？
- 抽象之后，每个接入方还是要写 70% 的定制代码吗？如果是，抽象的价值何在？

### 向前兼容

- 未来增加新能力时，接入方要不要改？
- 已有的 X 个平台能力之间会不会冲突？
- 接入方要花多少时间学习这个平台？

---

## 数据 / AI 类需求

**特点**：容易被"AI 魅力"裹挟、数据依赖强、效果不可控。

### 数据可行性

- 训练数据从哪来？够吗？标注过吗？
- 线上预测的数据分布和训练数据一致吗？
- 冷启动怎么办？第一天没数据的时候降级方案是什么？

### 效果保障

- 效果指标是什么？达不到怎么办？
- 人工规则能不能达到 80% 的效果？如果能，先用规则 + 少量 AI 会不会更合适？
- 失败 case 有多丑？用户能容忍吗？

### AI-slop 风险

- 这个 AI 生成的内容会不会充满废话（空话套话、内容没意义）？
- 有没有"人工审核"的环节？能不能关掉 AI 直接用规则？

---

## 业务变更类（非新功能）

**特点**：最容易"顺便改一下搞出大事故"。

### 影响面评估

- 这个改动影响多少用户？现存数据怎么迁移？
- 有没有**静默**影响——用户不知道但行为变了？
- 灰度策略是什么？1% → 10% → 50% → 100% 的推进节奏？

### 向后兼容

- 老数据还能工作吗？
- 客户端老版本怎么办？
- API 变了吗？调用方收到通知了吗？

### 回滚

- 如果上线后发现问题，多快能回滚？
- 回滚是"一键"还是"要半天"？
- 有没有数据不可回滚（schema 变更、数据迁移）？

---

## 通用反 AI-slop 检测问题

当你的需求描述里出现以下关键词时，额外追问（这些词通常是"没想清楚"的信号）：

| 关键词 | 追问 |
|---|---|
| "赋能" | 具体赋能什么？量化。 |
| "提效" | 提升多少？对比基线。 |
| "体验" | 哪个场景的体验？用户具体看到什么变化？ |
| "智能化" | 这个"智能"用规则能做 80% 吗？ |
| "闭环" | 哪些环节没闭？闭上了又怎样？ |
| "打通" | 现在为什么不通？打通了谁用？ |
| "协同" | 谁和谁协同？协同方愿意接入吗？ |
| "抓手" | 你想抓什么？手在哪？ |
| "对齐" | 需要谁对齐？对不齐怎么办？ |

## 提问时机决策树

```
用户刚说出需求
    │
    ▼
强度 5/5 必选问题：
  ├─► "不做会怎样？"（必要性）
  ├─► "谁用？多少人？"（用户画像）
  └─► 任选一个"角度三-五"的问题
    │
    ▼
用户回答了 → 追问具体数字/场景
  │
  │ 用户提供了具体数字
  ▼
判断数字是否合理：
  ├─► 合理 + 能说清 → 继续深挖边界
  └─► 不合理 / 含糊 → 用反例压测
    │
    ▼
核心需求稳定 → 切换 2/5
```

## 极端情况处理

### 用户很强势，拒绝被挑战

```
用户：这个需求必须做，老板定的。

你：
好。老板是直接对你说的，还是听谁说了之后定的？
如果是后者，他听到的信息完整吗？
如果是前者，他定了"做"，还是定了"做什么程度"？

如果是必做项，那讨论重点不是"做不做"，而是"做多少才算完成"。
来聊边界。
```

**原则**：不硬刚"必做"，但保留"做什么程度"的讨论空间。

### 用户已经想得很清楚

```
用户：不用问了，我已经想明白了，用户画像是 xxx，价值是 xxx，边界是 xxx。

你：
好。那我问三个验证性问题（如果都答得上来，直接进入下一阶段）：
1. [角度一的一个尖锐问题]
2. [角度四的一个尖锐问题]
3. [落地现实的一个尖锐问题]
```

**原则**：用 3 个核心问题**快速验证**，答得上来就放过。

### 用户想直接跳到写代码

```
用户：别废话了，我们直接开始写代码吧。

你：
可以。但在写代码前我必须核对 3 件事（会很快）：
1. [核心需求一句话]
2. [用户画像 + 量级]
3. [明确的 out-of-scope]

如果这 3 件你都能一句话回答，我们就开始。
如果任何一件含糊，先别写——后面返工代价更大。
```

**原则**：用**最小成本**的验证阻止盲目开工。
