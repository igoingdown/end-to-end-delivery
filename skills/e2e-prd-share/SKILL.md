# E2E PRD Share —— PRD 分享到飞书

## 定位

PRD 定稿后的**分享**动作。把本地的 Markdown PRD 让团队看到。

**MVP 边界**：

- ✅ 发 Markdown 文件附件到话题/群
- ✅ 同时发一张"PRD 摘要卡片"（便于快速浏览）
- ❌ 不做 Markdown → 飞书文档的转换（后续版本 `e2e-prd-share-v2`）

---

## 输入与输出

**输入**：

- PRD Markdown 文件路径（来自 `prd-generation` 的产出）
- 目标（话题/群/@人）

**输出**：

- 飞书话题中的消息卡片 + 附件
- 如果需要，提供共享链接

## 步骤详解

### 步骤 1：解析 PRD 提取摘要

从 Markdown PRD 中提取：

- **一句话说明**：从"一句话说明"章节
- **关键日期**：从"发布计划"章节的 M1-M5 里提取
- **Must 清单**：从"做什么（What）-> 核心能力（Must）"章节
- **负责人**：从 frontmatter 或"作者"字段
- **链接**：如果 PRD 里有飞书文档链接、BITS 链接

**解析失败** → 跳过该字段，其他字段继续。

### 步骤 2：生成摘要卡片

**卡片模板**：

```json
{
  "type": "interactive",
  "header": {
    "title": "📋 PRD · 用户分层规则自助配置",
    "theme": "blue"
  },
  "body": [
    {
      "type": "text",
      "content": "**一句话**：给运营同学一个自助配置用户分层规则的后台，替代手工 Excel 分层。"
    },
    {
      "type": "fields",
      "fields": [
        { "label": "负责人", "value": "@zhangsan" },
        { "label": "计划交付", "value": "2026-05-15" },
        { "label": "影响仓库", "value": "3 个" }
      ]
    },
    {
      "type": "divider"
    },
    {
      "type": "text",
      "content": "**Must 核心能力**：\n- 运营能通过 UI 配置 RPN 分层规则\n- 系统实时校验规则语法\n- 支持查看过去 30 天变更历史"
    },
    {
      "type": "note",
      "content": "完整 PRD 见下方附件"
    }
  ],
  "actions": [
    { "text": "查看完整 PRD", "type": "primary", "url": "[附件 URL]" }
  ]
}
```

### 步骤 3：上传 Markdown 文件

调 `feishu-cli-msg` 的**文件发送**能力：

```
让 feishu-cli-msg 上传文件：
- file_path: <PRD-xxx-v1.0-20260419.md>
- file_type: markdown
- target: <话题/群 ID>
```

上传后得到**文件 key**（飞书内部的文件引用）。

### 步骤 4：发卡片 + 附件

**组合消息**：

- 先发摘要卡片
- 再发文件附件（通过 file_key）

飞书显示效果：

```
[摘要卡片]
---
📎 PRD-用户分层规则自助配置-v1.0-20260419.md
```

## 消息粒度

### 选项 A：卡片 + 附件（默认）

见上面的步骤。**两条消息**。

优势：卡片便于快速浏览，附件是完整版本。

### 选项 B：只发卡片 + 链接

如果 PRD 已经上传到某个可访问的位置（Git 仓库、共享盘），可以不发文件：

```
卡片 + "查看 PRD" 按钮 → 跳到 URL
```

优势：文件版本可以持续更新（不像附件是快照）。

### 选项 C：只发文件（低调模式）

小改动 PRD，或者仅给 1-2 人 review，不需要卡片：

```
feishu-cli-msg 直接发文件 + 简短说明
```

### 默认选项

本 skill **默认选 A**。用户可指定其他。

## 和其他 skill 的协同

### 输入来自

- `prd-generation` —— PRD Markdown 文件
- `e2e-dev-task-setup`（可选）—— BITS 任务链接

### 输出给

- 飞书话题/群（最终归宿）

### 和 `e2e-progress-notify` 的区别

| Skill | 作用 |
|---|---|
| `e2e-prd-share` | 分享 PRD 文档本身（有 Markdown 附件） |
| `e2e-progress-notify` | 通用进度通知（可能提到 PRD 但不是主体） |

PRD 定稿时的典型调用顺序：

```
1. e2e-prd-share → 发 PRD 文件和摘要到话题
2. e2e-progress-notify → 通知 @PM @QA "PRD 好了，请 review"
```

## 失败处理

### 失败 A：PRD 文件不存在

- 告知主 Agent
- 建议用户先跑 `prd-generation`

### 失败 B：文件太大（> 20 MB）

- Markdown PRD 不应该这么大
- 检查 PRD 是否意外包含了大图片
- 建议用户精简

### 失败 C：目标话题找不到

- 要求用户明确指定话题 ID 或群名

### 失败 D：飞书 API 限流

- 等待 30 秒重试
- 仍失败 → `BLOCKED`

## 反 AI-slop 规范

### 禁用模式

- ❌ 把整个 PRD 正文塞进卡片（太长、难读）
- ❌ 摘要自己二次创作（偏离原 PRD）
- ❌ 加无关内容（"请大家认真阅读"这种废话）
- ❌ 发完就自动 @ 所有人

### 正确模式

- ✅ 卡片只摘要关键信息，原文在附件
- ✅ 摘要**直接引用**原 PRD 的句子，不改写
- ✅ 精确 @ 相关人
- ✅ 让 PM/相关人**自己判断**需要不需要细读
